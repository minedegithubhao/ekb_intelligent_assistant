# RAG 核心功能：检索流程设计（整理后）

## 0. 整理说明

本文档基于原始文档《检索流程.md》整理而成，重点做以下处理：

- 保留原有 1 到 9 步主流程，不调整整体检索策略。
- 将前置搭建、版本管理、规则配置、检索参数、证据格式等内容结构化。
- 对实现时容易产生歧义的位置补充说明，并统一命名。
- 对阈值、提示词、规则集合等内容给出推荐初始值，后续可通过日志和评测继续调优。

本文档不包含以下内容的详细设计：

- 管理端页面交互。
- 用户权限与登录流程。
- 知识库导入页面。
- 参数配置接口。
- 动态检索计划的具体算法。

## 1. 核心流程总览

```text
用户提问原句
  ↓
原句归一化
  ↓
预设规则判断
  ├─ 打招呼：固定回复
  ├─ 越界：固定回复
  └─ 转人工：固定回复
  ↓
FAQ 快速检索
  ├─ 完全匹配：直接返回 FAQ
  └─ 未命中：继续
  ↓
追问判断
  ├─ 是追问：历史消息摘要 + 最近消息 + LLM 改写
  └─ 不是追问：继续
  ↓
检索策略 + 动态计划
  └─ 当前版本暂不实现，直接 pass
  ↓
变体生成
  ├─ 原始提问或改写后提问
  ├─ 规则变体
  └─ LLM 变体
  ↓
FAQ 混合检索
  ├─ 高置信：直接返回 FAQ 或严格基于 FAQ 回答
  ├─ 中置信：保留 FAQ evidence，继续查文档
  └─ 低置信：丢弃 FAQ evidence，继续查文档
  ↓
文档库混合检索
  ├─ 命中文档证据：回填父块，生成 doc evidence
  └─ 低于阈值：返回“知识库未找到可靠依据”
  ↓
最终生成
```

## 2. 前置搭建

### 2.1. 向量知识库

向量知识库选用 Milvus 2.5+，使用 Milvus 内置 BM25 能力支持稀疏检索。

知识库按内容类型拆分为两类 collection：

| collection 类型 | 存储内容 | 主要用途 |
| --- | --- | --- |
| FAQ collection | FAQ 问答对 | 快速命中标准问答，处理明确问题 |
| 文档 collection | 知识文档分块 | 支撑复杂问题、长尾问题和补充依据 |

向量类型：

| 向量类型 | 实现方式 | 说明 |
| --- | --- | --- |
| 稠密向量 | bge-m3 | 用于语义相似召回 |
| 稀疏向量 | Milvus 2.5+ 内置 BM25 | 用于关键词和字面匹配召回 |

### 2.2. FAQ 分块策略

FAQ 按完整问答对作为一个逻辑块。

原设计要求：FAQ 只对“问题”做稠密向量和稀疏向量。

落地时可以统一理解为：

- 检索向量字段主要来自 FAQ 问题文本。
- 返回结果和最终回答时携带完整问答对。
- FAQ 去重主键使用 `faq_id`。

建议 FAQ 基础字段：

| 字段 | 说明 |
| --- | --- |
| `faq_id` | FAQ 唯一标识 |
| `question` | 标准问题 |
| `answer` | 标准答案 |
| `dense_vector` | 问题文本的稠密向量 |
| `sparse_vector` | 问题文本的 BM25 稀疏向量 |
| `kb_version` | 所属知识库版本 |
| `enabled` | 是否启用 |
| `metadata` | 标签、来源、业务分类等扩展信息 |

### 2.3. 文档分块策略

文档按 markdown 标题递归分类，采用父子块结构。

原设计要求：文档需要准备稠密向量和稀疏向量。

父子块职责：

| 分块类型 | 职责 |
| --- | --- |
| 子块 child chunk | 用于精准召回和 rerank |
| 父块 parent chunk | 用于最终提供给 LLM 的完整上下文 |

检索时先检索子块，再根据 `parent_id` 回填父块：

```text
child chunk hybrid search
  ↓
child rerank
  ↓
根据 parent_id 回填 parent chunk
  ↓
得到文档 evidence
```

建议文档基础字段：

| 字段 | 说明 |
| --- | --- |
| `child_chunk_id` | 子块唯一标识 |
| `parent_id` | 父块标识 |
| `title_path` | markdown 标题路径 |
| `child_content` | 子块内容 |
| `parent_content` | 父块内容，可单独存储或按 parent_id 查询 |
| `dense_vector` | 子块稠密向量 |
| `sparse_vector` | 子块稀疏向量 |
| `source_doc_id` | 来源文档标识 |
| `kb_version` | 所属知识库版本 |
| `metadata` | 文档来源、章节、业务标签等 |

## 3. Collection 版本管理

### 3.1. 版本管理原则

结合 MySQL `kb_versions` 表对 Milvus collection 做版本管理。

核心原则：

- MySQL 管理当前 active version 和版本状态。
- Milvus collection 使用版本号命名。
- Redis 缓存当前线上版本，减少频繁查库。
- 新版本自动校验通过后再切换为线上版本。
- 旧版本先进入 standby 保留，不要立刻删除。
- 超过保留期后再归档或 drop。

### 3.2. `kb_versions` 表

原始字段要求：

- `kb_version`
- 是否启用
- 创建日期
- 类型，FAQ / 文档

整理后的建议字段如下，实际实现可按项目已有表结构裁剪：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `kb_version` | 知识库版本号 |
| `collection_name` | Milvus collection 名称 |
| `type` | FAQ / DOC |
| `status` | building / ready / active / standby / archived / dropped / failed |
| `enabled` | 是否启用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `activated_at` | 启用时间 |
| `remark` | 备注 |

如 FAQ 和文档版本需要成组发布，也可以增加批次字段，例如 `release_id`，用于把同一次构建的 FAQ collection 和 DOC collection 关联起来。

### 3.3. 状态机

| 状态 | 说明 |
| --- | --- |
| `building` | 正在构建 |
| `ready` | 构建完成，待发布 |
| `active` | 当前线上使用 |
| `standby` | 上一个可回滚版本 |
| `archived` | 已归档，不在线使用 |
| `dropped` | 已删除 |
| `failed` | 构建失败 |

### 3.4. 推荐流转

```text
building
  ↓ 构建成功并通过自动校验
ready
  ↓ 发布
active
  ↓ 被新版本替换
standby
  ↓ 超过保留期
archived
  ↓ 确认无需保留
dropped
```

异常流转：

```text
building
  ↓ 构建失败或校验失败
failed
```

### 3.5. 发布和回滚

发布流程：

1. 构建新版本 Milvus collection。
2. 写入 FAQ 和文档向量数据。
3. 创建或加载稠密、稀疏检索能力。
4. 自动校验 collection 可用性。
5. 校验通过后将版本置为 `ready`。
6. 发布时将新版本置为 `active`。
7. Redis 刷新当前 active version。
8. 旧 active 版本改为 `standby`。

回滚流程：

1. 查找仍处于 `standby` 的上一版本。
2. 将 Redis 中 active version 切换回旧版本。
3. MySQL 同步更新版本状态。
4. 观察检索是否恢复正常。

注意：被 drop 的 Milvus collection 不能直接回滚，只能通过源数据或备份重新构建。

### 4.1. 关键词规则配置文件

该配置文件支持在管理端前端界面中查看、修改、更新。

配置集合：

| 集合 | 示例 |

说明：

- 以上关键词集合前期不需要细化，先配置少量示例即可。
- 后续可单独完善和补充关键词。
- 如果一个问题同时命中多个集合，需要在实现时明确优先级。当前文档按原章节顺序描述：打招呼、越界、转人工、FAQ 快速检索。

| --- | --- |

为了便于实现和日志记录，统一使用以下命名：

| 名称 | 说明 |
| --- | --- |
| `raw_question` | 用户提问原句 |
| `normalized_question` | 归一化后的问题 |
| `standalone_question` | 追问改写后的独立问题；非追问时可等于 `normalized_question` |
| `rule_variant_question` | 规则变体 |

## 6. 详细检索流程

### 6.1. 用户提问原句

输入：

```text
raw_question
```

要求：

- 保留用户原始提问，用于日志、审计、最终生成参考。
- 后续归一化和改写不要覆盖原始问题。

### 6.2. 对原句做归一化

对 `raw_question` 做归一化，得到 `normalized_question`。

原设计包含：

- 半角、全角标点符号全部统一为半角。
- 全部改为小写。
- 去除所有空格。
- 标准化时间。

补充说明：

- 归一化结果主要用于规则匹配、FAQ 精确匹配和检索 query 生成。
- 时间标准化需要明确标准格式，例如 `yyyy-MM-dd`、`yyyy-MM` 或业务约定格式。
- “去除所有空格”应按原设计实现；如果后续遇到英文产品名、错误码、命令等场景，再单独确认是否需要例外规则。

### 6.3. 通过预设规则判断是否可以直接回答

查询“关键词规则配置文件”，判断是否可以直接固定回复。

#### 6.3.1. 打招呼

条件：

```text
normalized_question 包含打招呼关键词集合中任意一个词
```

动作：

```text
直接返回：你好我是 xxx 知识库 AI，有什么可以帮您？
```

#### 6.3.2. 越界

条件：

```text
normalized_question 包含越界关键词集合中任意一个词
```

动作：

```text
直接返回：很抱歉，我是 xxx 知识库 AI，您可以向我提问 xxx 相关的问题
```

#### 6.3.3. 转人工

条件：

```text
normalized_question 包含转人工关键词集合中任意一个词
```

动作：

```text
直接返回：现在为您转接人工客服，请稍后...
```

### 6.4. FAQ 快速检索

该步骤用于处理明确、短句、标准化程度较高的 FAQ 问题。

#### 6.4.1. 规则判断

同时满足以下条件才进入 FAQ 快速检索：

1. 查询“关键词规则配置文件”，`normalized_question` 包含 FAQ 检索关键词集合中任意一个词。
2. 提问字符长度小于等于 48 个字符。

其中 48 保存在“检索参数配置文件”中，支持管理员端动态修改。

如果不满足以上全部条件，跳过当前步骤，进入“是否为追问”。

#### 6.4.2. 开始检索

检索方式：

```text
FAQ collection hybrid search
```

初始参数：

```text
稠密检索 top-k = 3
稀疏检索 top-k = 3
稠密向量模型 = bge-m3
稀疏向量 = Milvus BM25
```

命中判断：

1. 遍历 FAQ 检索结果。
2. 将用户问题与检索结果中的 FAQ 问题原文做比较。
3. 如果完全匹配，则将该 FAQ 问答对作为结果，直接返回用户。
4. 如果没有完全匹配，则继续进入下一步。

补充说明：

- 精确匹配建议统一比较口径，例如都使用归一化后的问题进行比较；如果必须比较原文，需要确保标点、大小写、空格不会导致误判。
- 该步骤只是快速命中优化，未命中不代表 FAQ 不能回答，后续仍会进入 FAQ 混合检索。

### 6.5. 是否为追问

是否为追问通过规则判断。

同时满足以下条件时，认为当前问题可能是追问：

1. 当前会话存在历史消息记录。根据当前会话 id 查询历史消息。
2. 提问字符长度小于等于 10 个字。

其中 10 保存在“检索参数配置文件”中，支持管理员端动态修改。

如果不符合规则，跳过当前步骤，进入“检索策略 + 动态计划”。

#### 6.5.1. 消息改写

处理历史消息：

1. 统计当前会话下所有历史消息。
2. 最近 8 条消息保留原文。
3. 8 条以前的消息交给 LLM 改写生成摘要。
4. 摘要设置字符上限，推荐初始值为 800，保存在“检索参数配置文件”中。

其中 8 保存在“检索参数配置文件”中，支持管理员端动态修改。

追问改写目标：

- 替换追问中的指代词。
- 将依赖上下文的问题改写成可独立检索的问题。
- 不改变用户原始意图。
- 不引入历史消息中不存在的新约束。

输出：

```text
standalone_question
```

如果不是追问：

```text
standalone_question = normalized_question
```

传给下一步的上下文：

```text
历史摘要 + 最近 8 条消息 + standalone_question
```

### 6.6. 检索策略 + 动态计划

当前版本暂不实现。

代码中保留占位逻辑：

```text
pass
```

当前流程直接进入“变体生成”。

### 6.7. 变体生成

通过用户提问生成 1 条规则变体和 1 条 LLM 变体。

加上用户原本的提问，共计 3 条 query。

```text
query_variants = [
  standalone_question,
  rule_variant_question,
  llm_variant_question
]
```

#### 6.7.1. 规则变体

查询“变体提示词配置文件”。

如果用户提问中出现配置文件中的同义词、别名或非标准说法，则统一替换为官方学名词，生成一条规则变体。

示例：

```text
笔记本电脑：[laptop、lap top、笔记型电脑]
CPU：[CPU、中央处理器、中央处理单元]
Wi-Fi：[Wi-Fi、WIFI、无线网络]
```

输出：

```text
rule_variant_question
```

#### 6.7.2. LLM 变体

使用 qwen-max 大模型，提前准备好提示词。

如果第五步做了消息改写：

```text
输入 = 摘要 + 最近 8 条消息 + standalone_question
```

如果没有消息改写：

```text
输入 = standalone_question
```

输出：

```text
llm_variant_question
```

LLM 变体要求：

- 只生成 1 条变体。
- 保持原问题意图不变。
- 不扩展成多个问题。
- 不引入知识库以外的新信息。
- 输出应适合作为检索 query。

### 6.8. 混合检索

拿到 3 条 query 后，先做 FAQ 混合检索。

如果 FAQ 未达到设定阈值，则继续做文档库混合检索。

#### 6.8.1. FAQ 混合检索

初始参数：

```text
fetch_k = 20
k = 20
dense_weight = 0.5
sparse_weight = 0.5
```

流程：

```text
3 个 query 分别 hybrid search top20
  ↓
三路结果按 faq_id 去重
  ↓
RRF / 加权融合
  ↓
取 top20
  ↓
rerank
  ↓
取 top3
  ↓
判断 FAQ 置信度
```

FAQ 置信度判断：

| 置信度 | 处理方式 |
| --- | --- |
| 高置信 | 直接返回 FAQ，或让 LLM 严格基于 FAQ 回答 |
| 中置信 | 保留 FAQ top3，继续查文档 |
| 低置信 | 丢弃 FAQ，继续查文档 |

补充说明：

- 3 条 query 的结果需要按 `faq_id` 去重，避免同一 FAQ 多次进入最终 evidence。
- RRF / 加权融合用于合并多路 query 结果。
- FAQ 高置信阈值和中置信阈值需要配置化，推荐初始值分别为 0.85 和 0.65。
- 如果选择“让 LLM 严格基于 FAQ 回答”，提示词必须限制 LLM 只能使用 FAQ 答案内容。

#### 6.8.2. 文档库混合检索

注意：文档库采用父子块结构。

初始参数：

```text
fetch_k = 50
k = 20
dense_weight = 0.7
sparse_weight = 0.3
```

流程：

```text
3 个 query 分别 hybrid search 子块 top20
  ↓
三路结果按 child_chunk_id 去重
  ↓
RRF / 加权融合
  ↓
取 top20
  ↓
child rerank
  ↓
取 top3 / top5 子块
  ↓
判断文档阈值
  ↓
根据 parent_id 回填父块
  ↓
得到文档 evidence
```

补充说明：

- 子块用于检索，父块用于最终生成上下文。
- 如果多个子块属于同一个父块，回填父块时需要去重。
- 文档证据阈值需要配置化，推荐初始值为 0.55。
- 如果所有文档结果都低于阈值，则认为知识库未找到可靠依据。

### 6.9. 最终生成

根据 FAQ 置信度和文档 evidence 构造最终输入。

#### 6.9.1. FAQ 中置信

处理方式：

```text
FAQ evidence + 文档 evidence
  ↓
final rerank
  ↓
基于最终 evidence 生成回答
```

说明：

- FAQ 中置信时不直接返回 FAQ。
- 需要继续查文档，把 FAQ 和文档证据合并后再做一次 final rerank。

#### 6.9.2. FAQ 低置信

处理方式：

```text
只使用文档 evidence
```

如果文档 evidence 也全部低于阈值：

```text
返回：知识库未找到可靠依据
```

#### 6.9.3. FAQ 高置信

原设计在 FAQ 检索阶段已经给出处理方式：

```text
直接返回 FAQ 或让 LLM 严格基于 FAQ 回答
```

如果使用 LLM 生成，建议约束：

```text
只能基于命中的 FAQ 标准答案回答；
不得补充知识库中没有的信息；
不得改变 FAQ 答案结论。
```

## 7. Evidence 结构

为了便于 FAQ、文档、final rerank 和最终生成统一处理，建议将证据统一封装为 evidence item。

### 7.1. FAQ Evidence

```text
type: faq
id: faq_id
title: FAQ 问题
content: FAQ 答案
score: rerank 分数
source: FAQ 知识库
kb_version: 知识库版本
metadata: 业务标签、适用范围等
```

### 7.2. 文档 Evidence

```text
type: doc
id: child_chunk_id 或 parent_id
title: 文档标题或标题路径
content: 回填后的父块内容
score: rerank 分数
source: 来源文档
kb_version: 知识库版本
metadata: parent_id、child_chunk_id、title_path 等
```

### 7.3. Final Evidence

最终交给 LLM 的证据集：

```text
final_evidence = [
  evidence_item_1,
  evidence_item_2,
  ...
]
```

要求：

- evidence 数量需要受 `final_evidence_top_k` 控制。
- evidence 内容需要保留来源，便于审计。
- 如果 evidence 来自 FAQ，应优先遵循 FAQ 的标准答案。
- 如果 evidence 不足，应返回未找到可靠依据，而不是让 LLM 自由发挥。

## 8. 返回策略汇总

| 场景 | 返回策略 |
| --- | --- |
| 命中打招呼关键词 | 返回固定欢迎语 |
| 命中越界关键词 | 返回固定越界提示 |
| 命中转人工关键词 | 返回转人工提示 |
| FAQ 快速检索完全匹配 | 直接返回 FAQ 问答对 |
| FAQ 混合检索高置信 | 直接返回 FAQ，或严格基于 FAQ 生成 |
| FAQ 混合检索中置信 | 保留 FAQ top3，继续查文档，合并后 final rerank |
| FAQ 混合检索低置信 | 丢弃 FAQ，继续查文档 |
| 文档证据达标 | 基于文档 evidence 生成回答 |
| 文档证据也低于阈值 | 返回“知识库未找到可靠依据” |

## 10. 推荐初始值与落地约定

以下内容作为当前版本的推荐初始值和落地约定，后续可根据日志、人工评测和 golden question 结果继续调优。

| 配置项 | 推荐值 / 推荐处理 |
| --- | --- |
| FAQ 高置信阈值 | `faq_high_conf_threshold = 0.85` |
| FAQ 中置信阈值 | `faq_middle_conf_threshold = 0.65` |
| 文档证据阈值 | `doc_evidence_threshold = 0.55` |
| 历史摘要最大字符数 | `history_summary_max_chars = 800` |
| 最终 evidence 数量 | `final_evidence_top_k = 6` |
| rerank 模型 | `bge-reranker-v2-m3` |
| LLM 改写提示词 | 使用固定模板 `rewrite_prompt_v1` |
| LLM 变体提示词 | 使用固定模板 `query_variant_prompt_v1` |
| LLM 最终生成提示词 | 使用固定模板 `evidence_answer_prompt_v1`，强约束只能基于 evidence 回答 |
| 规则命中优先级 | 转人工 > 越界 > 打招呼 > FAQ 快速检索 |
| 空格归一化细节 | 中文语境空格删除；英文产品名、命令、错误码中的必要空格保留 |
| FAQ 精确匹配口径 | 使用归一化后的问题做完全匹配 |
| 旧版本保留期 | standby 版本保留 30 天，且至少保留最近 1 个可回滚版本 |

## 11. 当前版本实现边界

当前版本建议先实现固定流程：

1. 问题归一化。
2. 关键词规则直接回复。
3. FAQ 快速检索和完全匹配。
4. 追问识别和消息改写。
5. 规则变体和 LLM 变体。
6. FAQ 三 query 混合检索、去重、融合、rerank。
7. 文档子块三 query 混合检索、去重、融合、rerank。
8. 文档父块回填。
9. FAQ / 文档 evidence 合并和最终生成。

暂不实现：

```text
检索策略 + 动态计划
```

该模块代码中保留 `pass`，后续可单独扩展。
